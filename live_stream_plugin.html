直播心率补光插件

直播场景是 rPPG 的另一个更轻、更容易被感知的落地方向：主播本来就需要摄像头和补光灯，如果补光设备或直播插件能顺手提供心率 Overlay，就可以把“稳定补光”和“非接触心率检测”合成一个产品体验。

服务对象

第一版插件优先服务三类人：

  • 主播和内容创作者：在直播中展示实时心率，用于运动、游戏、带货、情绪挑战、健康类内容。

  • 直播运营和 MCN：把心率作为互动素材，增强直播间可看性和话题性。

  • 补光灯/摄像头硬件厂商：把 rPPG 做成差异化卖点，而不是只卷亮度、色温和外观。

第一阶段：OBS + B站

当前仓库提供了第一阶段原型：demo/live-heart-light-plugin/。

它采用 Python 常驻采集服务 + OBS Browser Source + B站推流：

  1. model_server.py 独立打开摄像头，持续读取 30fps 视频帧。

  2. 服务端调用当前仓库的 rppg.Model('FacePhys.rlap') 和 update_frame()，复用仓库自带 BlazeFace/ONNX 人脸检测与追踪。

  3. 服务端统一输出 HR、SQI、采集状态、无人脸状态和 Overlay 状态。

  4. camera.html 作为 OBS 摄像头画面源。

  5. overlay.html 作为 OBS 心率 Overlay。

  6. 主播使用 OBS 推流到 B站测试直播间。

这个阶段不做原生 OBS C++ 插件，也不接入 B站直播姬内部插件能力。目标是先验证“只要本地服务存在，全直播时段就能持续监测”，并避免普通网页切后台导致信号断链。

为什么不再让网页采集

普通网页不能保证后台持续采集。切换标签页、最小化窗口或页面进入后台时，浏览器可能节流 requestAnimationFrame、摄像头帧和定时器。rPPG 又依赖连续、稳定、带真实时间戳的人脸颜色序列，因此网页采集一旦中断，模型窗口就需要重新积累。

第一阶段改为后端常驻采集后，控制台页面只是调试面板；关闭控制台、切屏、最小化浏览器，都不会中断摄像头输入和模型推理。

补光策略

这个方向真正有意思的地方不是“摄像头也能测心率”，而是补光灯可以主动帮助 rPPG：

  • 测量期间锁定亮度和色温，避免灯光闪烁导致信号污染。

  • 发现脸部过暗或过曝时，提示主播调整补光亮度和角度。

  • 心率 Overlay 可以轻微脉动，但真实补光不应跟着快速闪烁。

  • 后续硬件版可以把补光灯、摄像头、rPPG 推理和直播 Overlay 做成一体。

当前能力

  • 使用项目现有 FacePhys 模型输出 HR/SQI。

  • 服务端独立采集摄像头，网页不是采集生命线。

  • Overlay 直接轮询服务端状态，不依赖控制台页面写 localStorage。

  • 摄像头停止、无人脸或 SQI 偏低时，Overlay 不继续沿用旧 BPM。

  • 保留低 SQI 预览状态，但直播输出阈值保持 SQI >= 0.38。

  • OBS 中添加 camera.html 和 overlay.html 两个 Browser Source 即可测试。

  • 支持上传已有视频离线分析 HR/SQI，用于复盘素材或无摄像头时演示模型能力。

迭代路线

第一阶段：OBS + B站原型

  • Python 服务常驻采集摄像头。

  • OBS Browser Source 显示摄像头画面和心率 Overlay。

  • 控制台支持上传视频，单独跑 FacePhys 离线分析，不污染直播采集窗口。

  • 完成 30-60 分钟测试直播，记录 input FPS、SQI、BPM 输出率、无人脸次数、CPU/内存。

第二阶段：直播工程化

  • 支持选择摄像头设备、分辨率和帧率。

  • 加入本地服务启动器、异常恢复和看门狗。

  • 支持 OBS Virtual Camera 或虚拟摄像头输出，为抖音、快手、小红书等闭源直播工具做准备。

第三阶段：补光灯硬件联动

  • 控制亮度、色温和测量锁定模式。

  • 对接 USB/Bluetooth/Wi-Fi 补光灯。

  • 为游戏、运动、带货、教学等直播类型提供不同 Overlay 模板。

产品边界

直播插件展示的是“直播互动心率”或“状态氛围”，不是医疗监测。页面需要明确提示结果仅用于内容互动和体验展示，不用于诊断、治疗或健康决策。

现有 rPPG 模型不能原样直接嵌入纯网页。当前仓库的主模型依赖 Python、Keras/JAX、自定义层、.weights.h5 权重和 jax.jit 推理；浏览器无法直接执行这条链路。因此第一阶段采用本地模型服务接入现有模型。
