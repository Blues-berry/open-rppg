产品形态探索：面向高危岗位的 rPPG 安全备份

Open-rppg 更适合被产品化为“非接触式生理状态安全备份”，而不是单独承担主要监测或医疗诊断职责。它的价值在于利用已有摄像头，在不额外穿戴设备的情况下，为高风险作业岗位提供一层低打扰、可持续、可解释的辅助预警。

核心定位

产品应当围绕三个原则设计：

  • 先验证环境，再输出结果：rPPG 对光照、运动、遮挡、摄像头质量和肤色反射变化敏感。系统必须先确认当前采集环境可信，否则只输出“不可用/需复测”，不能给出看似精确但不可靠的数值。

  • 作为安全备份，而非主监测手段：在驾驶、列车巡检、飞行、精密流水线、无尘车间等场景中，rPPG 只作为现有安全系统、人工流程、穿戴设备或岗位规程的补充证据。

  • 关注趋势和风险状态，而非单次数值：产品层不应把心率、HRV 等指标包装成确定性诊断，而应输出“采集可信度、状态趋势、异常置信度、建议动作”。

可信采集校验

正式检测前需要有一个类似人脸识别活体检测的“环境与采集可信度校验”流程。这个流程的目的不是证明人是谁，而是证明后续 rPPG 信号有足够好的采集条件。

1. 设备与画面自检

系统启动后先检查摄像头和画面质量：

  • 摄像头分辨率、帧率、延迟是否达到模型要求。

  • 人脸是否稳定出现在画面中，ROI 面积是否足够，面部是否过暗、过曝或被口罩、防护镜、帽檐严重遮挡。

  • 环境光是否稳定，是否存在频闪、强背光、剧烈色偏或屏幕反光。

  • 自动曝光、自动白平衡是否导致颜色通道持续漂移；必要时提示固定机位、补光或锁定曝光参数。

2. 主动交互校验

借鉴人脸识别中的眨眼、张嘴、多色验证，可以设计轻量的主动挑战：

  • 眨眼/张嘴/轻微转头：确认当前检测到的是实时人脸，并验证面部关键区域没有被静态照片、视频回放或遮挡物替代。

  • 多色光校验：通过屏幕或小型指示灯依次显示红、绿、蓝、白等颜色，检查摄像头对颜色变化的响应是否正常，并识别明显的色彩失真、曝光锁死或通道异常。

  • 静止采样窗口：要求用户保持 10-30 秒相对稳定姿态，系统评估 BVP 波形、频域峰值、SQI、丢帧率和人脸跟踪稳定性。

只有当以上校验通过后，系统才进入正式监测。校验失败时，产品应给出明确原因，例如“光照不稳定”“人脸 ROI 过小”“信号质量不足”，而不是输出低可信度心率。

3. 信号可信度门控

运行中也要持续门控检测结果：

  • SQI 低于阈值时暂停输出生理状态，仅显示采集质量问题。

  • 心率结果超出生理合理范围或短时间剧烈跳变时，标记为异常采集而非异常人体状态。

  • 当面部跟踪丢失、姿态变化过大、遮挡增加或光照突变时，自动降级到“观察中/不可判定”。

  • 只有连续多个时间窗都满足质量条件，才触发风险趋势判断。

适合落地的岗位

优先选择“事故代价高、人员姿态相对固定、已有摄像头基础、可以接受周期性校验”的岗位。

场景

适配原因

驾驶员、工程车辆操作员

车载辅助安全模块，和 DMS、制动/语音提醒系统联动

产品形态

1. 岗前可信准入

面向上岗前 30-60 秒的快速检测：

  • 完成人脸在位、活体动作、多色光、静止采样和 SQI 检查。

  • 生成本次班次的基础状态参考值。

  • 输出“通过、需重测、需人工确认”三类结果。

  • 只把 rPPG 作为准入流程中的一个辅助项，不单独决定人员能否上岗。

2. 班中安全哨兵

面向工作过程中的低打扰监测：

  • 持续采集，但只在信号质量达标时更新状态。

  • 关注趋势偏离、持续高负荷、疑似疲劳或突发异常，而非每秒展示心率。

  • 异常先进入“二次确认”：语音提示、屏幕确认、短暂停留、班组长复核，避免单次误报直接干预业务。

  • 与已有安全系统联动时，只提供辅助信号和置信度，不直接替代制动、停机、调度等主控制逻辑。

3. 边缘盒子或 SDK

更现实的商业形态可以是两层：

  • 边缘盒子：接入摄像头，在本地完成画面校验、rPPG 推理、SQI 门控和事件生成，适合工厂、车队、轨交场站。

  • 软件 SDK/API：提供 采集质量 -> 生理指标 -> 风险状态 -> 事件回调 的标准接口，嵌入 DMS、工位管理、安监平台或训练复盘系统。

4. 管理端看板

管理端不应做成“实时监控每个人心率”的界面，而应聚焦安全运营：

  • 当前可用设备、采集质量、离线/遮挡/光照异常。

  • 按班次、岗位、设备统计的风险事件数量和人工确认结果。

  • 误报、漏报、复测率、有效采集率等产品质量指标。

  • 默认不存储原始视频，只保留必要的事件摘要、质量评分和审计记录。

运行闭环

一个可落地的系统不应只做“摄像头采集 -> 心率输出”，而应设计成闭环状态机：

这个闭环的关键是“不可判定”必须是一个正式状态。对于 rPPG 产品来说，承认当前环境不能可靠测量，比输出一个漂亮但不可信的数字更重要。

风险分级

建议把告警设计成渐进式，而不是二元化报警：

等级

触发条件

产品动作

L0 不输出

SQI 不足、人脸丢失、强光/遮挡/运动过大

提示采集条件问题，不判断人体状态

L1 观察

指标轻微偏离，但持续时间短或置信度一般

L2 提醒

轻量提醒本人确认状态

L3 复核

其中 L4 不应由 rPPG 单独触发。对于驾驶、飞行、列车、工厂停机等高影响动作，rPPG 只能提供辅助证据，最终应由主安全系统、人工确认或既有规程决定。

岗位策略包

不同岗位不应共用一套阈值和交互。更合理的产品方式是预置“岗位策略包”，让部署人员按场景选择和微调。

策略包至少需要配置这些参数：

  • 最低帧率、最低 ROI 面积、允许的头部姿态范围。

  • 初始采样窗口和运行中滑动窗口长度。

  • SQI 阈值、连续有效窗口数量、异常趋势持续时间。

  • 是否允许保存事件截图、是否保存原始视频、保存时长。

  • 提醒对象、升级路径、人工确认入口。

SDK/API 形态

如果将 Open-rppg 封装成产品 SDK，可以把输出分成三层，而不是只暴露心率：

  1. 采集质量层：判断当前画面是否可用。

  2. 生理指标层：在质量达标时输出 HR、HRV、BVP、SQI。

  3. 安全事件层：结合岗位策略输出风险状态和建议动作。

示例事件结构：

{
  "worker_id": "hashed-or-local-id",
  "station_id": "line-a-03",
  "timestamp": "2026-06-30T10:20:30+08:00",
  "capture_quality": {
    "usable": true,
    "sqi": 0.86,
    "face_track_stability": 0.93,
    "lighting_stability": 0.91
  },
  "signals": {
    "hr_bpm": 82.4,
    "hrv_status": "within_shift_baseline",
    "window_seconds": 20
  },
  "risk": {
    "level": "L1",
    "reason": "mild_trend_deviation",
    "confidence": 0.62,
    "recommended_action": "observe"
  }
}

这个接口表达的重点是：当 capture_quality.usable 为 false 时，调用方不应该读取或展示 signals 作为可靠结论。

试点验证设计

产品试点可以拆成三个问题逐步验证：

1. 能不能稳定采

先不追求告警，只看真实岗位中的采集可用性：

  • 每天每个岗位有多少时间窗通过质量门控。

  • 哪些失败来自光照、遮挡、姿态、摄像头或人员移动。

  • 主动校验平均耗时和复测率是否可接受。

2. 测得准不准

在受控试点中对照接触式设备，但结论仍应表达为工程指标：

  • HR 与参考设备的误差分布。

  • 不同光照、肤色、年龄、眼镜、口罩、防护装备下的可用率。

  • 模型在运动、说话、低头、转头时的降级是否及时。

3. 提醒有没有价值

最后才验证风险事件：

  • 系统提醒后，人工确认是否认为有处理价值。

  • 提醒是否过多打扰正常工作。

  • 是否能发现“人没主动报告，但确实需要休息、复测或换岗”的情况。

  • 误报被如何解释，是否能通过采集质量、岗位策略或阈值调优降低。

试点早期的目标不是证明系统可以自动替代人，而是证明它能在不增加太多负担的情况下，稳定提供一层额外的安全观察。

Demo 原型

当前仓库提供了一个可运行的交互 Demo：demo/self-validating-camera/。它不把重点放在“摄像头测心率”，而是展示一个更有辨识度的产品逻辑：相机先完成自证可信，再开放生理状态和安全事件输出。

Demo 面向三类观众：

  • 一线岗位人员：看到系统不会在采集不可信时强行判断自己。

  • 安全管理者：看到风险事件来自证据链，而不是单个心率数字。

  • 系统集成方：看到 SDK/API 如何输出 capture_quality、signals 和 risk。

建议演示路径：

  1. 启动相机或进入模拟器，默认状态下心率输出被锁定。

  2. 运行可信校验，依次展示设备自检、活体动作、多色响应、静止采样和证书生成。

  3. 模拟光照突变或遮挡，展示系统进入“不可判定”，并拒绝输出生理结论。

  4. 模拟风险趋势，展示在采集可信时如何输出 L2/L3 安全事件。

  5. 展示 SDK JSON，让观众理解它如何接入 DMS、安监平台或工位系统。

这个 Demo 的创新点不是更炫的 UI，而是把 rPPG 包装成“可信采集协议 + 安全证据层”。它让产品从一个测量工具，变成一个知道自己什么时候可信、什么时候必须沉默的辅助安全系统。

MVP 路线

第一阶段可以从受控环境开始：

  1. 固定摄像头和补光条件，完成岗前可信采集校验。

  2. 使用 SQI、人脸跟踪稳定性、光照稳定性作为结果输出门槛。

  3. 在一个高危但流程可控的岗位做试点，例如固定工位操作员或驾驶模拟舱。

  4. 先验证“有效采集率、复测率、人工确认后的误报率、用户接受度”，再讨论自动干预。

第二阶段再进入班中监测：

  1. 增加滑动窗口趋势判断，避免单点报警。

  2. 接入岗位系统或安监平台，只推送风险事件和置信度。

  3. 建立人工确认闭环，把确认结果回流到阈值和场景策略中。

第三阶段扩展到多岗位：

  1. 为驾驶、巡检、飞行训练、精密制造分别配置场景策略。

  2. 针对口罩、防护镜、强光、夜间、震动等典型环境建立专门的不可用判定。

  3. 建立隐私、合规、审计和数据留存策略，避免产品被误用为医疗诊断或过度劳动监控。

成功指标

产品验证应优先看这些指标：

  • 有效采集率：在真实岗位中，有多少时间窗能通过环境和信号质量门控。

  • 复测率：岗前校验中需要重复采样的比例。

  • 误报率和确认率：系统提示的风险事件中，有多少被人工确认有处理价值。

  • 响应收益：系统提示是否帮助提前发现疲劳、身体不适、注意力下降或设备环境问题。

  • 用户接受度：员工是否认为它低打扰、可解释、不会被滥用于绩效监控。

边界与风险

这类产品必须把边界写清楚：

  • 不用于医疗诊断，不宣称可替代心电、血氧、血压或专业体检。

  • 不在低质量采集条件下输出确定数值。

  • 不把单次心率异常直接等同于危险行为或健康异常。

  • 不应单独作为处罚、停岗、保险、绩效评价依据。

  • 需要明确告知、授权、数据最小化、可审计和本地优先处理。

最稳妥的产品叙事是：rPPG 不是“判断人是否安全”的裁判，而是高风险岗位里的一层额外传感器。它先证明自己看得清、信号可信，再在关键时刻提醒人和系统多看一眼。
